# 90-Day Onboarding Plan: Azure Infrastructure Manager

**Created for:** New Azure Infrastructure Manager (Neurodivergent Cloud Engineer)  
**Status:** New Hire - Starting from Scratch  
**Duration:** 90 Days (3 months)  
**Date:** May 2026

---

## Document Overview

This plan is designed with neurodivergent needs in mind. Features include:
- **Clear structure** with defined phases
- **Specific, measurable tasks** (no ambiguity)
- **Regular check-in points** for feedback and adjustment
- **Flexibility built-in** for varying energy levels and needs
- **No surprises** - all expectations listed upfront
- **Time buffers** for learning and questions

---

## How to Use This Plan

1. **Print or save** a copy for yourself
2. **Set calendar reminders** for key dates
3. **Schedule weekly check-ins** with your direct manager
4. **Mark off completed items** as you go
5. **Communicate early** if timelines need adjustment
6. **Ask questions** - there are NO stupid questions during onboarding

# PHASE 1: Orientation & Foundation (Days 1-30)

## Week 1: Welcome & Systems Setup

- [ ] Receive equipment (laptop, phone, badge)
- [ ] Set up email and accounts

### Tuesday-Friday (Days 2-5)

# PHASE 3: Contributing & Ownership (Days 61-90)

## Week 9: First Improvement Project

### Choose One Project
Pick ONE of your identified improvements. Good candidates:
- Fix a critical security issue
- Improve backup procedures
- Update documentation
- Optimize costs
- Implement monitoring gaps

### Project Phases
1. **Plan (Day 1-2):** Document exact steps
2. **Review (Day 2-3):** Have manager/senior admin review plan
3. **Test (Day 3-5):** Test in non-production if possible

5. **Verify (Day 6-7):** Confirm it worked

# Neurodivergent-Friendly Supports

## Built-In Accommodations

### 1. Frequent Check-ins
- Weekly meetings provide structure and feedback
- Regular check-ins reduce anxiety about performance
- Manager can adjust expectations based on reality

### 2. Clear Expectations
- Every task has specific deliverables
- No ambiguous "figure it out yourself" situations
- Checklists help with task management

### 3. Documentation Focus
- Everything is documented (reduces need to remember)
- Creating documentation helps process and retain information
- Written records prevent miscommunication

### 4. Structured Learning
- Guided tours before independent exploration
- Deep dives one topic at a time
- No information overload

### 5. Manageable Scope
- 30-day phases prevent overwhelm
- One project at a time during Phase 3
- Clear priorities reduce decision paralysis

## Ask For If Needed

### Communication Preferences
- Do you prefer written or verbal instructions?
- Do you need more time to process information?
- Do you prefer fewer meetings or more structure?
- What time of day do you work best?

### Environmental Needs
- Do you need a quiet workspace?
- Do you need breaks between meetings?
- Do you need written agendas before meetings?

### Task Management
- Do you prefer daily task lists?
- Do you need task reminders?
- Do you prefer smaller tasks or larger projects?
- Do you need detailed step-by-step procedures?

### Communication Patterns
- Should interruptions be minimized?
- Preferred communication method (email, chat, meeting)?
- How much notice do you need for meetings?

## Imported Runbook Enhancements (conservative runbook additions)

Purpose: Add a stability-first runbook layer emphasizing read-only discovery, stakeholder mapping, documented change discipline, and clear deliverables. Keep this section as a living checklist and update weekly.

Days 0–30: Learn & Observe (additions)
- Complete access provisioning checklist: Entra tenant ID, Azure Portal, CLI/PowerShell, PIM/eligible roles, repository access. Verify you can view but not modify production resources.
- Use read-only inventory tooling and exports: `az group list`, `az resource list`, Azure Resource Graph queries, subscription exports, and Entra ID reports. Save outputs to your personal baseline document.
- Shadow a change window, deployment, or incident response where possible (observe, do not change).

Days 31–60: Understand & Contribute (additions)
- Own 2–4 low-risk, well-scoped tasks (tag remediation, alert tuning, documentation updates, minor IaC reviews) with explicit rollback plans and peer review.
- Pair with a senior engineer on at least one production change or incident to learn approval and rollback processes.

Days 61–90: Own & Improve (additions)
- Drive 1–2 scoped improvements end-to-end: proposal, design review, testing, implementation, documentation, and handover. Measure before/after metrics where applicable.
- Deliver a concise 90-day retrospective with validated recommendations and remaining gaps.

Expanded Discovery Questions (use as checklist while building baseline)
- Tenant & subscriptions: tenant ID, verified domain, subscriptions list and purposes, management group hierarchy, assigned policies and inherited RBAC, subscription owners and billing contacts.
- Identity: hybrid sync (Azure AD Connect) status, Conditional Access policies, PIM usage and break-glass accounts, custom RBAC roles, Entra security groups mapping to RBAC.
- Networking: VNets/subnets inventory, topology (hub-spoke/mesh/Virtual WAN), ExpressRoute/VPN circuits, NSGs/ASGs/Azure Firewall/third-party appliances, private endpoints/Private Link usage.
- Security & compliance: assigned Azure Policy initiatives, Microsoft Defender for Cloud posture, Key Vault/secret rotation, applicable compliance frameworks and evidence locations.
- Backup & DR: backup mechanisms, RPO/RTO targets, last DR test and results.
- Automation & IaC: provisioning tools (Bicep/Terraform/ARM), pipeline systems (Azure DevOps/GitHub Actions), Git repo locations for IaC.
- Monitoring & incident: Log Analytics workspaces, alerting and Action Groups, incident tracking tool, on-call rotation, post-incident review storage.
- Cost: budgets/alerts, primary cost drivers, tagging strategy and owners.

Deliverables (explicit additions)
- "Azure Environment Baseline – [Your Name] – 30-Day Observations" (inventory exports, diagrams, unanswered questions).
- Stakeholder contact map (names, roles, escalation preferences).
- At least one updated operational runbook and one dependency map for a production system by Day 60.
- Completed improvement with before/after documentation and handover notes by Day 90.

Signals of Success (explicit additions)
- You can locate production workloads, owners, and primary dependencies independently.
- Required access works (view-only for prod until approved for changes).
- At least two contributions accepted (tickets/docs/changes) and low-risk.
- At least one improvement implemented, documented, and producing measurable value or risk reduction.

Operating Principles (must-follow additions)
- Document first, change second. If documentation is missing or wrong, update documentation as part of the work.
- Every change requires a rollback plan, testing steps, communication plan, and peer/lead review.
- Stability and predictability beat speed during onboarding; prefer incremental, well-understood improvements.
- When in doubt, write the question down, propose who might know, and follow up.

Quick Read-only Command Examples
- `az login --tenant <tenant-id>`
- `az account list --output table`
- `az group list --output table`
- `az resource list --subscription <id> --output table`
- Use Azure Resource Graph Explorer for cross-subscription queries.

Keep these additions as an active checklist and integrate them into your weekly 1:1 updates.
